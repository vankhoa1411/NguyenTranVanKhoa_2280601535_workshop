#### Overview

In this step, we will configure the **Prisma Schema** for the **DocuMind AI** system.

Prisma ORM simplifies database interactions for the Node.js/TypeScript backend. Instead of writing SQL statements manually, Prisma allows developers to define the database schema in a single `schema.prisma` file, and then generate the Prisma Client. This enables type-safe database queries and migrations.

In DocuMind AI, the Prisma Schema will define the core tables: `User`, `Document`, `OCRResult`, `AIAnalysis`, `ChatHistory`, `Notification`, and `AuditLog`.

---

#### Objectives of This Step

Upon completing this step, you will:

* Install Prisma in the backend project.
* Configure the PostgreSQL datasource connection.
* Create the core models for DocuMind AI.
* Design relationships between User, Document, OCRResult, and AIAnalysis.
* Prepare the schema to execute database migrations in the next step.

---

#### The Role of Prisma in DocuMind AI

In the **DocuMind AI** system, Prisma is used to:

* Manage the database schema.
* Generate and track database migrations for PostgreSQL.
* Perform database queries in the backend.
* Store uploaded document metadata.
* Store OCR text extracted via Amazon Textract.
* Store AI Analysis results generated by Gemini/OpenAI.
* Manage notifications and audit logging.
* Minimize database query runtime errors using TypeScript types.

#### Step 1: Install Prisma

In the backend directory, install Prisma:

```bash
npm install prisma --save-dev
npm install @prisma/client
```

Initialize Prisma:

```bash
npx prisma init
```

This command generates the following workspace structure:

```text
prisma/
└── schema.prisma

.env
```

---

#### Step 2: Configure PostgreSQL Datasource

Open the file:

```text
prisma/schema.prisma
```

Configure the datasource:

```prisma
generator client {
  provider = "prisma-client-js"
}
 
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

Prisma retrieves the connection string from the environment:

```env
DATABASE_URL="postgresql://docmind:docmind_password@localhost:5432/docmind"
```

#### Step 3: Design the User Model

The `User` model stores user account details.

Example model:

```prisma
model User {
  id           String   @id @default(uuid())
  email        String   @unique
  passwordHash String?
  fullName     String?
  avatarUrl    String?
  role         UserRole @default(USER)
  isActive     Boolean  @default(true)
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
 
  documents     Document[]
  notifications Notification[]
  auditLogs     AuditLog[]
  chatHistory   ChatHistory[]
}
 
enum UserRole {
  USER
  ADMIN
}
```

In the system, the `role` parameter determines User or Admin privileges.

---

#### Step 4: Design the Document Model

The `Document` model stores metadata for documents uploaded to S3.

```prisma
model Document {
  id        String         @id @default(uuid())
  userId    String
  fileName  String
  fileType  String
  fileSize  Int
  s3Bucket  String
  s3Key     String
  status    DocumentStatus @default(PENDING)
  createdAt DateTime       @default(now())
  updatedAt DateTime       @updatedAt
 
  user       User         @relation(fields: [userId], references: [id], onDelete: Cascade)
  ocrResult  OCRResult?
  aiAnalysis AIAnalysis?
  chatHistory ChatHistory[]
}
 
enum DocumentStatus {
  PENDING
  UPLOADED
  QUEUED
  PROCESSING
  OCR_COMPLETED
  AI_ANALYZING
  COMPLETED
  FAILED
}
```

Key attributes:

| Field       | Description                                  |
| ----------- | -------------------------------------------- |
| `s3Bucket`  | Name of the target S3 bucket                 |
| `s3Key`     | Object key representing the file path in S3  |
| `status`    | Processing stage of the document             |

#### Step 5: Design the OCRResult Model

The `OCRResult` model stores the text content extracted by Amazon Textract.

```prisma
model OCRResult {
  id         String   @id @default(uuid())
  documentId String   @unique
  provider   String   @default("AWS_TEXTRACT")
  rawText    String
  createdAt  DateTime @default(now())
 
  document Document @relation(fields: [documentId], references: [id], onDelete: Cascade)
}
```

Each document maps to a single main OCR result.

---

#### Step 6: Design the AIAnalysis Model

The `AIAnalysis` model stores analysis results from Gemini or OpenAI.

```prisma
model AIAnalysis {
  id                   String   @id @default(uuid())
  documentId           String   @unique
  provider             String
  model                String
  documentType         String?
  summary              String?
  entities             Json?
  keywords             Json?
  importantInformation Json?
  confidence           Float?
  createdAt            DateTime @default(now())
 
  document Document @relation(fields: [documentId], references: [id], onDelete: Cascade)
}
```

Using `Json` types allows flexible storage for lists of entities, keywords, and key parameters.

---

#### Step 7: Design the ChatHistory Model

The `ChatHistory` model stores Q&A history for AI Chat/RAG.

```prisma
model ChatHistory {
  id         String   @id @default(uuid())
  userId     String
  documentId String
  provider   String
  model      String
  question   String
  answer     String
  sources    Json?
  createdAt  DateTime @default(now())
 
  user     User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  document Document @relation(fields: [documentId], references: [id], onDelete: Cascade)
}
```

#### Step 8: Design the Notification Model

The `Notification` model handles notifications for Users and Admins.

```prisma
model Notification {
  id         String   @id @default(uuid())
  userId     String?
  roleTarget String?
  type       String
  severity   String
  title      String
  message    String
  metadata   Json?
  isRead     Boolean  @default(false)
  createdAt  DateTime @default(now())
  readAt     DateTime?
 
  user User? @relation(fields: [userId], references: [id], onDelete: Cascade)
}
```

Example notifications:

* Document uploaded successfully.
* OCR completed.
* AI analysis completed.
* AI provider failed.
* Admin security alert.

---

#### Step 9: Design the AuditLog Model

The `AuditLog` model tracks key operational events.

```prisma
model AuditLog {
  id          String   @id @default(uuid())
  userId      String?
  action      String
  target      String?
  description String?
  ipAddress   String?
  userAgent   String?
  metadata    Json?
  createdAt   DateTime @default(now())
 
  user User? @relation(fields: [userId], references: [id], onDelete: SetNull)
}
```

Audit logs should be created for:

* Logins and logouts.
* Document uploads.
* Document deletions.
* User role updates.
* Administrator actions.
* AI provider processing failures.
* Security events.

---

#### Step 10: Validate the Schema

After defining the schema, format the file contents:

```bash
npx prisma format
```

Validate the schema for configuration errors:

```bash
npx prisma validate
```

If no errors are found, Prisma will output that the schema is valid.
